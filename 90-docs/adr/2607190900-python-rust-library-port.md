# ADR-2607190900: Add a public Python + Rust library port to this repo

## Status

Accepted (2026-07-19).

## Context

The sqrt-checkpoint KV-residency algorithm existed only as private,
internal Clojure (`.cljc`) inside `gftdcojp/cloud-murakumo`. This repo
(`com-junkawasaki/sqrt-space-kv`) held the paper, benchmarks, and arXiv
metadata, but zero source code -- there was no way for the wider
Python/Rust community to use the technique outside this monorepo's
Clojure serving stack.

Placement was initially considered under `kotoba-lang` (the org this
monorepo's generic libraries default to), but the owner asked for it to
stay under `com-junkawasaki` instead, since the technique isn't specific
to the kotoba-lang substrate/ecosystem. Since GitHub does not allow two
repos with the same name in one org, and this repo already holds the name
`sqrt-space-kv` under `com-junkawasaki`, the library is added directly
into this existing, already-public repo rather than as a second repo.

## Decision

Hand-port `sqrt_space.cljc` and the pure subset of `kv_runtime.cljc`
(excluding `kv_policy.cljc`'s vLLM/mlx-moe serving glue and Clojure-only
CLI/file-I/O code) into independent, dependency-free `python/` and
`rust/` packages in this repo, both named `sqrt-space-kv`.

Conformance to the canonical `.cljc` is enforced via
`tests/fixtures/golden.json`, generated by a new fixture-dump script in
`cloud-murakumo` (`src/cloud_murakumo/sqrt_space_kv_fixtures.clj`,
`:sqrt-kv-fixtures` alias) and vendored (copied by hand, not CI-synced --
`cloud-murakumo` is private `gftdcojp`, this repo is public
`com-junkawasaki`, so a public repo's CI cannot reach into the private one
without a cross-org token).

No headline memory-reduction percentages appear in the library-specific
docs (`python/README.md`, `rust/README.md`, module docstrings) --
per `90-docs/adr/2607182800-sqrt-space-kv-mla-composability-maturity-review.edn`
in the superproject, external announcement of those headline numbers is
gated pending unresolved evidence (host-paging measurably slows decode:
2.2-3.7x in measured configs). The library ships as neutral, honestly
caveated utility code; the top-level repo `README.md` (pre-existing
research reporting, not new promotional content) is unaffected.

### Rejected alternatives

- **Rust-core + PyO3/FFI bindings.** This workspace's monorepo has
  deliberately moved away from Rust-core-plus-generated-bindings setups
  (deleted Rust+PyO3 workspaces in favor of per-language hand ports; see
  `90-docs/adr/2607010930-clj-wgsl-migration.edn`'s citation of the
  `forbidden-authority-files` convention). Reintroducing that pattern here
  would be inconsistent with established practice for a small,
  already-dependency-free math library.
- **WASM-as-single-source** (complete the `.kotoba` port, emit via
  `kotoba wasm emit`, consume from Python/Rust via `wasmtime`). Adds a
  wasm-runtime dependency to consumers of a ~550-line, already
  dependency-free math library -- not proportionate for something this
  small.
- **Separate `kotoba-lang/sqrt-space-kv` repo.** Would have avoided the
  west.yml basename question entirely, but the owner's explicit direction
  was `com-junkawasaki`, and GitHub's same-org-same-name constraint makes
  "separate new repo, same name" impossible anyway -- the two live options
  were "new repo, different name" or "same repo as the paper." The latter
  was chosen.

## Consequences

- `python/` (PyPI, not yet published) and `rust/` (crates.io, not yet
  published) packages added to this repo; both zero-dependency.
- `LICENSE` (MIT) added -- neither this repo nor `cloud-murakumo` had one
  before.
- Manual (not CI-automated) sync process between `cloud-murakumo`'s
  canonical `.cljc` and this repo's golden fixture (see
  `CONTRIBUTING.md`), since the two repos are in different GitHub orgs
  with different default visibility.
- No west.yml change needed for this repo's own registration (existing
  `sqrt-space-kv` entry unchanged); only the normal pin advance after
  these commits land.

### Open questions

- Cross-language float tolerance for the golden-fixture attention
  comparison was set conservatively (`1e-6` vs. the fixture's Clojure
  values, `1e-9` within-language) without extensive tuning -- revisit if
  it proves too loose or too tight in practice.
- Versioning/sync cadence if `cloud-murakumo`'s `.cljc` changes after
  this repo's `v0.1.0` is unspecified beyond "manual, documented in
  CONTRIBUTING.md" -- automated cross-repo sync would need a cross-org
  access decision, not attempted here.
